# Project Constitution: Physical AI Documentation Website

**Project Name**: Physical AI Documentation Website
**Platform**: Docusaurus 3.9.2
**Purpose**: A scalable, book-style documentation website for teaching robotics, digital twins, and physical AI through structured modules and chapters
**Constitution Version**: 1.1.0
**Ratification Date**: 2025-12-06
**Last Amended Date**: 2025-12-06

---

## Governance

### Amendment Procedure

1. **Proposal**: Any team member may propose a constitutional amendment by creating a pull request that:
   - Updates this constitution file
   - Documents the rationale in the PR description
   - Identifies affected templates and dependent artifacts
   - Provides a versioning justification (MAJOR, MINOR, or PATCH)

2. **Review**: Amendments require:
   - Review of impact on existing specifications, plans, and tasks
   - Validation that templates remain consistent
   - Confirmation that no active features are blocked

3. **Approval**: Amendments are approved when:
   - All affected templates are updated
   - The version number is incremented appropriately
   - The Sync Impact Report is completed

4. **Propagation**: After approval:
   - Update the LAST_AMENDED_DATE to the merge date
   - Notify team of changed principles
   - Update any active feature branches affected by changes

### Versioning Policy

- **MAJOR** (X.0.0): Backward-incompatible changes, principle removals, or redefinitions that require existing work to be revised
- **MINOR** (x.Y.0): New principles added, materially expanded guidance, or new mandatory sections
- **PATCH** (x.y.Z): Clarifications, wording improvements, typo fixes, non-semantic refinements

### Compliance Review

- All feature specifications MUST reference and comply with this constitution
- The `/sp.plan` command MUST include a "Constitution Check" section
- Architecture Decision Records (ADRs) MUST be created when decisions conflict with or extend these principles
- Quarterly reviews to ensure principles remain relevant and actionable

## Core Principles

### Principle 1: Book-First Content Architecture

**Statement**: All content MUST be organized as a hierarchical book structure with modules, chapters, and sections. Each module represents a major subject area (e.g., "Module 2: The Digital Twin") and contains multiple chapters as MDX files.

**Rationale**: A book-like structure provides intuitive navigation for learners, supports progressive disclosure of complex topics, and scales naturally as new modules and chapters are added. This structure mirrors how technical knowledge is traditionally organized and consumed.

**Requirements**:
- Content lives in `docs/module-{number}-{slug}/` directories
- Each module has an `index.mdx` introduction file
- Each chapter is a separate `.mdx` file with frontmatter metadata
- Module structure: `docs/module-2-digital-twin/{intro.mdx, chapter-1.mdx, chapter-2.mdx, ...}`
- Chapters MUST include: title, description, sidebar_position, and tags in frontmatter
- Autogenerated sidebars MUST be used via `sidebars.ts` configuration
- No manual sidebar maintenance - structure drives navigation

**Validation**:
- All content files follow the `docs/module-*/chapter-*.mdx` pattern
- Each MDX file has valid frontmatter with required fields
- Sidebar generation works without manual intervention
- Module index files provide clear learning objectives

---

### Principle 2: MDX-First Content with Reusable Components

**Statement**: All documentation content MUST be written in MDX format to support rich, interactive components while maintaining readability and version control. Custom React components MUST be reusable across chapters and modules.

**Rationale**: MDX combines the simplicity of Markdown with the power of React, enabling interactive code examples, diagrams, callouts, and embedded media. Reusable components ensure consistency and reduce duplication across hundreds of documentation pages.

**Requirements**:
- All content files use `.mdx` extension
- Custom components live in `src/components/` with clear naming
- Common components include:
  - `<CodeBlock>` - syntax-highlighted code with copy button
  - `<Callout>` - info/warning/danger boxes
  - `<Tabs>` - tabbed content for multi-language examples
  - `<VideoEmbed>` - responsive video players
  - `<ChapterProgress>` - reading progress indicators
  - `<InteractiveDiagram>` - SVG/Canvas-based diagrams
  - `<Quiz>` - self-assessment components
- Components MUST be typed with TypeScript interfaces
- Components MUST be documented with JSDoc comments
- No inline styles - use Tailwind/CSS modules
- Import components explicitly in MDX files

**Validation**:
- All MDX files compile without errors
- Components render correctly in dev and production builds
- Components are accessible (WCAG 2.1 AA)
- Component documentation exists in `src/components/README.md`

### Principle 3: Performance Budget and Optimization

**Statement**: The website MUST meet strict performance budgets to ensure fast loading on all devices and network conditions. Docusaurus optimizations MUST be leveraged fully.

**Rationale**: Documentation websites serve global audiences with varying network speeds and devices. Slow loading frustrates learners and reduces engagement. Performance is a feature, not an afterthought.

**Performance Budgets**:
- First Contentful Paint (FCP): < 1.8s
- Largest Contentful Paint (LCP): < 2.5s
- Time to Interactive (TTI): < 3.8s
- Cumulative Layout Shift (CLS): < 0.1
- First Input Delay (FID): < 100ms
- Total bundle size: < 300KB (initial load, gzipped)
- Lighthouse score: > 90 (Performance, Accessibility, Best Practices, SEO)

**Required Optimizations**:
- Enable Docusaurus production optimizations (minification, code splitting)
- Use `@docusaurus/plugin-ideal-image` for responsive images
- Lazy-load images and videos below the fold
- Preload critical fonts and assets
- Use service workers for offline caching (via `@docusaurus/plugin-pwa`)
- Implement route-based code splitting
- Tree-shake unused JavaScript
- Optimize custom components with React.memo()
- Compress static assets (gzip/brotli)
- Use CDN for static asset delivery

**Validation**:
- Run Lighthouse on every production build
- Monitor Core Web Vitals via Google Search Console
- Set up performance budgets in CI/CD (fail builds that regress)
- Test on 3G throttled networks

### Principle 4: Mobile-First Responsive Design

**Statement**: All UI components and layouts MUST be designed mobile-first and work flawlessly across all screen sizes. The persistent sidebar MUST adapt to a drawer navigation on mobile devices.

**Rationale**: Over 50% of documentation traffic comes from mobile devices. Mobile-first design ensures core content is prioritized and layouts degrade gracefully. Persistent navigation aids learning flow.

**Requirements**:
- Design for 320px minimum width (iPhone SE)
- Breakpoints: mobile (< 768px), tablet (768-1023px), desktop (≥ 1024px)
- Sidebar behavior:
  - Mobile: Hidden by default, accessible via hamburger menu (drawer)
  - Tablet: Toggle-able with state persistence
  - Desktop: Always visible, persistent during navigation
- Touch targets MUST be ≥ 44x44px (iOS guidelines)
- Text MUST be readable without zooming (16px minimum)
- Forms and inputs MUST be usable without horizontal scrolling
- Images and videos MUST be responsive (srcset, picture element)
- Navigation MUST be accessible via keyboard and screen readers

**Validation**:
- Test on real devices: iPhone SE, iPad, Android phone, Desktop
- Use Chrome DevTools device emulation
- Verify touch interactions work correctly
- Check viewport meta tag is set correctly
- No horizontal scrolling on any device

### Principle 5: Automated Testing for Content and Functionality

**Statement**: Critical functionality MUST be covered by automated tests. Content structure and links MUST be validated automatically. Tests are OPTIONAL for documentation content itself but REQUIRED for custom components and plugins.

**Rationale**: Automated tests prevent regressions, ensure links don't break, and validate that custom components work across updates. However, prose documentation content does not require unit tests.

**Testing Requirements**:
- **Component Tests** (REQUIRED):
  - All custom React components MUST have Jest + React Testing Library tests
  - Test component rendering, props, accessibility, and interactions
  - Coverage target: > 80% for components
- **Link Validation** (REQUIRED):
  - Use `@docusaurus/plugin-client-redirects` to catch broken internal links
  - Run link checker in CI/CD (e.g., `markdown-link-check`)
  - Fail builds on 404 errors
- **Build Tests** (REQUIRED):
  - Validate MDX compilation succeeds for all files
  - Check for TypeScript errors
  - Validate sidebar generation
- **Visual Regression Tests** (OPTIONAL):
  - Use Percy or Chromatic for visual diffs
  - Catch unintended CSS changes
- **Content Tests** (OPTIONAL):
  - Vale.sh for style guide enforcement
  - Spell checking in CI

**Validation**:
- All tests pass in CI/CD before merge
- No broken links in production
- Components are tested and documented

### Principle 6: Security and Safe Content Handling

**Statement**: All user-facing functionality MUST follow security best practices. External content (videos, iframes, images) MUST be sandboxed and validated.

**Rationale**: Documentation websites can be attack vectors via XSS, malicious iframes, or compromised dependencies. Secure defaults protect users and maintain trust.

**Security Requirements**:
- **Content Security Policy (CSP)**:
  - Define strict CSP headers in `docusaurus.config.ts`
  - Whitelist trusted domains for scripts, styles, images, frames
  - Use nonces for inline scripts
- **Dependency Management**:
  - Run `npm audit` in CI/CD, fail on high/critical vulnerabilities
  - Keep Docusaurus and dependencies up to date
  - Use Dependabot or Renovate for automated updates
- **Input Sanitization**:
  - Sanitize any user-generated content (comments, search inputs)
  - Use `DOMPurify` for HTML sanitization
- **HTTPS Only**:
  - Enforce HTTPS for all production deployments
  - Use HSTS headers
- **Iframe Sandboxing**:
  - Use `sandbox` attribute for embedded content
  - Restrict to `allow-scripts allow-same-origin` only when necessary
- **Access Control**:
  - If versioning includes private beta docs, use authentication plugins
  - Protect staging environments with basic auth

**Validation**:
- Run security audits in CI/CD
- Test CSP headers with `securityheaders.com`
- Verify no XSS vulnerabilities via manual testing

### Principle 7: SEO and Metadata Standards

**Statement**: All pages MUST include complete, accurate SEO metadata to maximize discoverability. Structured data MUST be added for rich search results.

**Rationale**: Documentation is only useful if learners can find it. Proper SEO ensures content ranks well in search engines and appears correctly when shared on social media.

**SEO Requirements**:
- **Page Metadata** (REQUIRED for all pages):
  - `title`: Unique, descriptive, < 60 characters
  - `description`: Clear summary, < 160 characters
  - `keywords`: Relevant terms (comma-separated)
  - `image`: Open Graph image (1200x630px)
- **Structured Data** (JSON-LD):
  - Use `@docusaurus/plugin-google-gtag` for analytics
  - Add `TechArticle` or `LearningResource` schema
  - Include breadcrumbs schema
- **Open Graph Tags**:
  - og:title, og:description, og:image, og:url, og:type
  - Twitter Card tags: twitter:card, twitter:title, twitter:description, twitter:image
- **Sitemap and Robots**:
  - Generate sitemap.xml automatically (built-in)
  - Configure robots.txt to allow crawling
  - Submit sitemap to Google Search Console
- **Canonical URLs**:
  - Set canonical URLs to avoid duplicate content
  - Use `@docusaurus/plugin-client-redirects` for URL changes
- **Performance**:
  - Fast loading times improve SEO rankings (see Principle 3)

**Validation**:
- Check metadata with Google Rich Results Test
- Validate Open Graph with Facebook Sharing Debugger
- Monitor rankings and CTR in Google Search Console
- Use Screaming Frog or Sitebulb for full-site audits

### Principle 8: Versioning-Ready Architecture

**Statement**: The project structure MUST support Docusaurus versioning from day one, even if not immediately used. Version snapshots MUST be immutable and clearly labeled.

**Rationale**: Documentation evolves as libraries and frameworks change. Versioning allows users to access docs for the specific version they're using while development continues on future versions.

**Versioning Requirements**:
- **Structure**:
  - Use Docusaurus built-in versioning (`docusaurus docs:version X.Y`)
  - Versions stored in `versioned_docs/version-X.Y/`
  - Version configuration in `versions.json`
- **Version Strategy**:
  - Current version: `docs/` (latest, unreleased)
  - Released versions: `versioned_docs/version-1.0/`, `version-1.1/`, etc.
  - Use semantic versioning for module releases
- **Navigation**:
  - Version dropdown in navbar (Docusaurus default)
  - Clearly label "Latest" vs "v1.0" vs "v2.0"
  - Add version banners for outdated docs
- **Content Management**:
  - NEVER edit `versioned_docs/` directly - versions are snapshots
  - Make changes in `docs/` for next version
  - Backport critical fixes with `docusaurus docs:version:patch`
- **Deprecation**:
  - Mark deprecated modules/chapters with callouts
  - Provide migration guides for breaking changes
  - Archive very old versions (keep last 3 major versions)

**Validation**:
- Version switching works correctly
- Old versions remain accessible and immutable
- No broken links between versions

### Principle 9: Accessibility (WCAG 2.1 AA Compliance)

**Statement**: All content and functionality MUST meet WCAG 2.1 Level AA accessibility standards. Accessibility is a core feature, not an optional enhancement.

**Rationale**: Accessible documentation serves all learners, including those using screen readers, keyboard navigation, or assistive technologies. Accessibility also improves usability for everyone.

**Accessibility Requirements**:
- **Semantic HTML**:
  - Use proper heading hierarchy (h1 → h2 → h3, no skipping)
  - Use semantic elements (nav, main, article, aside, footer)
  - Proper ARIA labels for interactive elements
- **Keyboard Navigation**:
  - All interactive elements accessible via Tab, Enter, Space, Arrow keys
  - Focus indicators visible (outline, ring)
  - Skip-to-content link at top of page
  - Keyboard shortcuts documented
- **Screen Reader Support**:
  - Alt text for all images (descriptive, not decorative)
  - ARIA labels for icons and buttons
  - Announce dynamic content changes (ARIA live regions)
  - Logical reading order
- **Color and Contrast**:
  - Text contrast ≥ 4.5:1 (WCAG AA)
  - Large text (≥ 18pt) contrast ≥ 3:1
  - Don't rely on color alone to convey meaning
  - Support dark/light modes with proper contrast in both
- **Forms and Controls**:
  - Labels for all inputs
  - Error messages descriptive and associated with fields
  - Required fields clearly marked
- **Media**:
  - Captions for videos (WebVTT)
  - Transcripts for audio content
  - Pause/stop controls for auto-playing media

**Validation**:
- Run axe DevTools or WAVE on all pages
- Test with screen readers (NVDA, JAWS, VoiceOver)
- Keyboard-only navigation testing
- Automated accessibility checks in CI (axe-core, pa11y)

### Principle 10: Maintainability and Code Quality

**Statement**: All custom code (components, plugins, config) MUST follow consistent standards to ensure long-term maintainability. Code reviews and linting are mandatory.

**Rationale**: Documentation websites are long-lived projects. Clean, well-documented code ensures future contributors can understand and extend the system.

**Code Quality Requirements**:
- **Linting and Formatting**:
  - ESLint for JavaScript/TypeScript (Airbnb or Standard config)
  - Prettier for code formatting (enforced via pre-commit hooks)
  - Stylelint for CSS/SCSS
  - Markdownlint for MDX files
- **TypeScript**:
  - Use TypeScript for all custom code (strict mode)
  - Type all component props and function signatures
  - Avoid `any` types (use `unknown` or proper types)
- **Code Structure**:
  - Components: `src/components/{ComponentName}/{ComponentName}.tsx`
  - Hooks: `src/hooks/use{HookName}.ts`
  - Utils: `src/utils/{utilName}.ts`
  - Types: `src/types/{typeName}.ts`
  - One component per file
- **Documentation**:
  - JSDoc comments for all public functions and components
  - README.md in each major directory explaining purpose
  - Component usage examples in Storybook or MDX docs
- **Git Workflow**:
  - Feature branches: `feature/{module-name}/{description}`
  - Conventional commits: `feat:`, `fix:`, `docs:`, `style:`, `refactor:`, `test:`, `chore:`
  - Require PR reviews before merging
  - Squash commits on merge
- **Dependencies**:
  - Minimize dependencies (avoid npm bloat)
  - Audit dependencies for security and maintenance
  - Document why each dependency is needed

**Validation**:
- Linting and type-checking pass in CI/CD
- Code review checklist enforced
- No console errors or warnings in production builds

**Deployment Requirements**:

  - **Staging**: `staging.docs.example.com` (develop branch)
  - **Preview**: PR-specific URLs (auto-generated)
- **Rollback**:
  - Keep last 10 production builds
  - One-click rollback in hosting platform
  - Document rollback procedure
- **Monitoring**:
  - Uptime monitoring (Pingdom, UptimeRobot)
  - Error tracking (Sentry)
  - Analytics (Google Analytics via `@docusaurus/plugin-google-gtag`)

**Validation**:
- Deployments succeed in < 5 minutes
- Zero-downtime deployments
- Rollback tested quarterly

### Principle 13: Advanced AI Perception and Simulation (Module 3)

**Statement**: Content covering NVIDIA Isaac Sim, Isaac ROS, and Nav2 MUST demonstrate hardware-accelerated perception pipelines, synthetic data generation workflows, and bipedal humanoid path planning. All examples MUST be reproducible on NVIDIA GPU hardware.

**Rationale**: Module 3 teaches advanced AI-driven perception and navigation using NVIDIA's Isaac platform, which requires GPU acceleration and specialized simulation environments. This module bridges the gap between digital twin simulation (Module 2) and real-world deployment by providing photorealistic training data and production-grade perception algorithms.

**Requirements**:
- **NVIDIA Isaac Sim**:
  - Photorealistic simulation environment for synthetic data generation
  - GPU-based raycasting for LiDAR, depth cameras, and RGB sensors
  - Physics-accurate simulation synchronized with ROS 2
  - Support for large-scale scene generation and randomization
  - Documentation MUST include minimum hardware requirements (RTX GPU, VRAM)
- **Isaac ROS (VSLAM and Perception)**:
  - Hardware-accelerated Visual SLAM (VSLAM) using NVIDIA CUDA
  - Real-time odometry and mapping for humanoid robots
  - Integration with ROS 2 navigation stack (Nav2)
  - GPU-optimized DNN inference for object detection and segmentation
  - Documentation MUST include performance benchmarks (FPS, latency)
- **Nav2 Path Planning**:
  - Bipedal humanoid-specific navigation strategies
  - Dynamic obstacle avoidance in complex environments
  - Integration with Isaac ROS localization
  - Cost maps and path planning algorithms for two-legged locomotion
  - Support for stair climbing, uneven terrain, and narrow passages
- **Synthetic Dataset Generation**:
  - Automated data collection workflows in Isaac Sim
  - Ground-truth annotations (bounding boxes, segmentation masks, depth)
  - Domain randomization for sim-to-real transfer
  - Export formats compatible with ML training pipelines (COCO, YOLO, TFRecord)
- **Content Structure** (Module 3):
  - `docs/module-3-ai-robot-brain/index.mdx` - Module introduction
  - `chapter-1-isaac-sim-setup.mdx` - Installation and environment setup
  - `chapter-2-synthetic-data.mdx` - Photorealistic data generation workflows
  - `chapter-3-isaac-ros-vslam.mdx` - Hardware-accelerated VSLAM implementation
  - `chapter-4-nav2-planning.mdx` - Path planning for bipedal humanoids
  - `chapter-5-integration.mdx` - End-to-end perception and navigation pipeline

**Hardware Requirements**:
- NVIDIA RTX GPU (RTX 3060 or higher recommended)
- CUDA 11.8+ and cuDNN 8.6+
- Ubuntu 20.04/22.04 (Isaac Sim and Isaac ROS compatibility)
- ROS 2 Humble or later
- 16GB+ RAM, 32GB+ recommended for large scenes

**Validation**:
- All Isaac Sim examples tested on minimum spec GPU (RTX 3060)
- Isaac ROS VSLAM achieves real-time performance (>= 30 FPS)
- Nav2 path planning examples work for bipedal gaits
- Synthetic datasets generate valid ground-truth annotations
- All code examples include hardware performance metrics

### Content Guidelines

- **Headings**: Use h2 (`##`) for main sections, h3 (`###`) for subsections
- **Code Blocks**: Use fenced code blocks with language identifiers
- **Links**: Use relative links for internal navigation (`./other-chapter`)
- **Images**: Store in `static/img/modules/module-N/` and reference with `/img/...`
- **Videos**: Prefer YouTube embeds (lazy loading) over self-hosted
- **Callouts**: Use for tips, warnings, and important notes
- **Quizzes**: Add at the end of chapters for self-assessment
- **Prev/Next**: Docusaurus auto-generates pagination based on sidebar order

## System Workflow Explanation

### How Docusaurus Works (End-to-End)

1. **Content Authoring**:
   - Write MDX files in `docs/module-{number}-{slug}/`
   - Add frontmatter for metadata and sidebar control
   - Import custom components as needed

2. **Build Process** (`yarn build`):
   - Docusaurus reads `docusaurus.config.ts` for site configuration
   - Parses MDX files and compiles them to React components
   - Generates sidebars from `sidebars.ts` and frontmatter
   - Applies theme (default or swizzled components from `src/theme/`)
   - Bundles JavaScript with Webpack (code splitting, tree shaking)
   - Optimizes CSS (minification, critical CSS extraction)
   - Generates static HTML for each page (SSG - Static Site Generation)
   - Creates `sitemap.xml`, `robots.txt`, and metadata
   - Outputs to `build/` directory

3. **Development Server** (`yarn start`):
   - Hot module reloading (changes reflect instantly)
   - Fast refresh for React components
   - Development-only warnings and debugging tools
   - Runs on `http://localhost:3000`

4. **Routing**:
   - Docusaurus uses file-based routing: `docs/module-2/chapter-1.mdx` → `/module-2/chapter-1`
   - `routeBasePath: '/'` makes docs the homepage
   - Client-side routing (React Router) for instant navigation

5. **Sidebar Generation**:
   - Reads `sidebars.ts` configuration
   - Matches file IDs from frontmatter to sidebar items
   - Generates nested navigation tree
   - Adds "Previous" and "Next" buttons based on sidebar order

6. **Theme and Components**:
   - Docusaurus provides default theme components (Navbar, Footer, DocPage, TOC)
   - "Swizzle" components to customize (eject and override)
   - Custom components imported directly in MDX files
   - CSS variables in `custom.css` override theme defaults

7. **Search** (Optional):
   - Algolia DocSearch: Crawls site, indexes content, provides instant search
   - Local search: Use `@easyops-cn/docusaurus-search-local` for client-side search

8. **Versioning**:
   - Run `yarn docusaurus docs:version 1.0` to snapshot current docs
   - Creates `versioned_docs/version-1.0/` and `versioned_sidebars/`
   - Version dropdown appears in navbar
   - Current `docs/` becomes the "Next" version

9. **Deployment**:
   - Build generates static files in `build/`
   - Deploy to any static host (Vercel, Netlify, GitHub Pages, AWS S3)
   - CI/CD pipeline automates: lint → test → build → deploy
   - Preview deployments for PRs (Vercel/Netlify auto-generate URLs)

10. **Performance**:
    - Static HTML loads instantly (SSG)
    - React hydrates after load for interactivity
    - Code splitting reduces initial bundle size
    - CDN caches assets globally
    - Service workers enable offline access (PWA plugin)

## Key Interactions

### User Journey (Reader)
1. Visits homepage (`docs.physical-ai.com`)
2. Sees Module 1 and Module 2 in sidebar
3. Clicks "Module 2: The Digital Twin"
4. Reads module introduction (`module-2-digital-twin/index.mdx`)
5. Clicks "Chapter 1 - Physics Simulation"
6. Reads chapter, interacts with code blocks and videos
7. Clicks "Next Chapter" to continue
8. Uses search to find specific topics
9. Toggles dark mode for evening reading
10. Website works offline (PWA cached)

### Contributor Journey (Developer)
1. Clones repository: `git clone https://github.com/your-org/physical-ai-docs.git`
2. Installs dependencies: `yarn install` (< 2 minutes)
3. Starts dev server: `yarn start` (opens browser automatically)
4. Creates new chapter: `docs/module-2-digital-twin/chapter-6-advanced-sensors.mdx`
5. Writes content using MDX with custom components
6. Saves file → Hot reload shows changes instantly
7. Commits with conventional commit: `git commit -m "docs: add chapter 6 to module 2"`
8. Pushes branch and opens PR
9. CI/CD runs: lint, build, lighthouse, link check
10. Preview deployment URL posted to PR for review
11. After approval, merges to main → Auto-deploys to production

- **User Feedback**: Analytics + surveys to identify gaps
- **Versioning**: Snapshot modules when they reach stable milestones
